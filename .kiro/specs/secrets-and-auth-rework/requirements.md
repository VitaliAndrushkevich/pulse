# Requirements Document

## Introduction

This feature reworks the existing "secrets" concept in Pulse into two clearly separated concerns:

1. **Monitor Auth Credentials** — Encrypted credentials (Bearer tokens, Basic auth, custom headers) that Pulse attaches to outbound health-check requests for specific monitors.
2. **API Tokens** — Tokens that users generate in Settings to access the Pulse API programmatically (CI/CD, automation, external integrations).

The current implementation stores both concepts in a single `secrets` table and presents them on one Settings page, creating confusion about their purpose and lifecycle. This rework introduces a dedicated `monitor_credentials` data model tied to individual monitors, while the existing `api_tokens` system remains in Settings with an improved UI that clearly communicates its purpose.

## Glossary

- **Credential_Store**: The backend service responsible for encrypting, storing, and retrieving monitor auth credentials using AES-256-GCM encryption at rest.
- **Monitor_Credential**: An encrypted authentication artifact (Bearer token, Basic auth pair, or custom header set) associated with a specific monitor and injected into outbound health-check requests.
- **API_Token**: A bcrypt-hashed token generated by a user to authenticate programmatic access to the Pulse API itself, managed in user Settings.
- **Checker**: A protocol-specific component (HTTP, TCP, UDP, WebSocket) that performs a single health-check request against a monitor target.
- **Auth_Type**: The category of authentication a Monitor_Credential represents: `bearer`, `basic`, or `header`.
- **Settings_Page**: The frontend page where users manage API Tokens for programmatic Pulse API access.
- **Monitor_Config_Page**: The frontend page (monitor create/edit form) where users configure Monitor_Credentials for a specific monitor.
- **Show_Once_Modal**: A modal dialog displayed immediately after credential or token creation that shows the raw secret value with a copy-to-clipboard button; once dismissed, the secret value is permanently non-retrievable.

## Requirements

### Requirement 1: Create Monitor Credential

**User Story:** As a Pulse user, I want to create an authentication credential for a specific monitor, so that Pulse can authenticate when performing health checks against protected endpoints.

#### Acceptance Criteria

1. WHEN a user submits a valid credential with an auth_type of `bearer`, a token value, and a monitor_id, THE Credential_Store SHALL encrypt the token value using AES-256-GCM and persist it associated with the specified monitor.
2. WHEN a user submits a valid credential with an auth_type of `basic`, a username, and a password, THE Credential_Store SHALL encrypt both the username and password using AES-256-GCM and persist them associated with the specified monitor.
3. WHEN a user submits a valid credential with an auth_type of `header`, a header name, and a header value, THE Credential_Store SHALL encrypt the header value using AES-256-GCM and persist it with the header name associated with the specified monitor.
4. WHEN a credential is created successfully, THE Credential_Store SHALL return the credential metadata (id, auth_type, created_at) without including any plaintext secret values in the API response.
5. WHEN a credential is created successfully, THE Monitor_Config_Page SHALL display the raw secret value (token, password, or header value) exactly once in a modal dialog with a copy-to-clipboard button.
6. WHEN the user dismisses the creation-success modal, THE Monitor_Config_Page SHALL discard the plaintext secret value permanently, making it non-retrievable through the UI or API.
7. IF a user submits a credential with a missing or invalid auth_type, THEN THE Credential_Store SHALL return a 400 error with a descriptive message indicating the valid auth_type values.
8. IF a user submits a credential referencing a monitor_id that does not exist, THEN THE Credential_Store SHALL return a 404 error.

### Requirement 2: List and Retrieve Monitor Credentials

**User Story:** As a Pulse user, I want to view the credentials configured for a monitor, so that I can audit and manage authentication settings.

#### Acceptance Criteria

1. WHEN a user requests the credentials for a specific monitor, THE Credential_Store SHALL return a list of credential metadata (id, auth_type, name, created_at, updated_at) without plaintext secret values.
2. WHEN a user requests credentials for a monitor that has no credentials configured, THE Credential_Store SHALL return an empty list.
3. THE Credential_Store SHALL include the header name for credentials of auth_type `header` in the metadata response.
4. THE Credential_Store SHALL include the username (but not the password) for credentials of auth_type `basic` in the metadata response.

### Requirement 3: Update Monitor Credential

**User Story:** As a Pulse user, I want to update an existing credential for a monitor, so that I can rotate secrets without recreating the credential.

#### Acceptance Criteria

1. WHEN a user submits an update to a credential with a new secret value, THE Credential_Store SHALL encrypt the new value using AES-256-GCM and replace the previous encrypted value.
2. WHEN a credential is updated successfully, THE Credential_Store SHALL update the `updated_at` timestamp and return the credential metadata without plaintext secret values.
3. IF a user submits an update referencing a credential_id that does not exist, THEN THE Credential_Store SHALL return a 404 error.

### Requirement 4: Delete Monitor Credential

**User Story:** As a Pulse user, I want to delete a credential from a monitor, so that Pulse stops sending authentication for that monitor's health checks.

#### Acceptance Criteria

1. WHEN a user deletes a credential, THE Credential_Store SHALL remove the encrypted credential from the database permanently.
2. WHEN a credential is deleted, THE Checker associated with the credential's monitor SHALL stop including that credential's authentication in subsequent health-check requests.
3. IF a user attempts to delete a credential_id that does not exist, THEN THE Credential_Store SHALL return a 404 error.

### Requirement 5: Inject Credentials into HTTP Health Checks

**User Story:** As a Pulse user, I want my HTTP monitors to automatically include configured credentials in their requests, so that Pulse can check protected endpoints.

#### Acceptance Criteria

1. WHEN the HTTP Checker executes a health check for a monitor with a `bearer` credential, THE Checker SHALL include an `Authorization: Bearer <token>` header in the outbound request.
2. WHEN the HTTP Checker executes a health check for a monitor with a `basic` credential, THE Checker SHALL include an `Authorization: Basic <base64(username:password)>` header in the outbound request.
3. WHEN the HTTP Checker executes a health check for a monitor with one or more `header` credentials, THE Checker SHALL include each configured header name and its decrypted value in the outbound request.
4. WHEN a monitor has multiple credentials configured, THE Checker SHALL include all of them in the outbound request.
5. IF the Credential_Store fails to decrypt a credential at check time, THEN THE Checker SHALL report the check as "down" with an error message indicating a credential decryption failure without exposing the encrypted value.

### Requirement 6: Inject Credentials into WebSocket Health Checks

**User Story:** As a Pulse user, I want my WebSocket monitors to include configured credentials in their upgrade handshake, so that Pulse can check authenticated WebSocket endpoints.

#### Acceptance Criteria

1. WHEN the WebSocket Checker executes a health check for a monitor with a `bearer` credential, THE Checker SHALL include an `Authorization: Bearer <token>` header in the WebSocket upgrade request.
2. WHEN the WebSocket Checker executes a health check for a monitor with a `basic` credential, THE Checker SHALL include an `Authorization: Basic <base64(username:password)>` header in the WebSocket upgrade request.
3. WHEN the WebSocket Checker executes a health check for a monitor with `header` credentials, THE Checker SHALL include each configured header in the WebSocket upgrade request.
4. IF the Credential_Store fails to decrypt a credential at check time, THEN THE Checker SHALL report the check as "down" with an error message indicating a credential decryption failure.

### Requirement 7: Credential Cascading Deletion

**User Story:** As a Pulse user, I want credentials to be automatically removed when I delete a monitor, so that orphaned encrypted data does not persist.

#### Acceptance Criteria

1. WHEN a monitor is deleted, THE Credential_Store SHALL delete all credentials associated with that monitor.
2. THE Credential_Store SHALL implement cascading deletion at the database level using a foreign key constraint with ON DELETE CASCADE.

### Requirement 8: API Token Management in Settings

**User Story:** As a Pulse user, I want to manage API tokens on a dedicated Settings page, so that I can create and revoke tokens for programmatic access to the Pulse API.

#### Acceptance Criteria

1. THE Settings_Page SHALL display a list of existing API tokens showing only metadata: name, creation date, last used date, and expiration status — never the token value.
2. WHEN a user creates a new API token, THE Settings_Page SHALL display the raw token value exactly once in a modal dialog with a copy-to-clipboard button and a warning that the token will not be shown again.
3. WHEN the user dismisses the token-creation modal, THE Settings_Page SHALL discard the plaintext token value permanently, making it non-retrievable through the UI or API.
4. WHEN a user revokes an API token, THE Settings_Page SHALL mark the token as revoked and cease accepting it for API authentication.
5. THE Settings_Page SHALL clearly label this section as "API Tokens" with a description explaining these tokens are for programmatic access to the Pulse API.
6. THE Settings_Page SHALL NOT display any monitor-specific credentials in this section.
7. THE Settings_Page SHALL NOT provide any mechanism to reveal, retrieve, or regenerate the value of an existing API token.

### Requirement 9: Remove Legacy Secrets Concept

**User Story:** As a Pulse user, I want the ambiguous "Secrets" terminology removed from the UI, so that the purpose of each credential type is clear.

#### Acceptance Criteria

1. THE Settings_Page SHALL remove the existing "Secrets" section and replace it with the "API Tokens" section.
2. THE Monitor_Config_Page SHALL provide the interface for managing Monitor_Credentials within the monitor create/edit form.
3. WHEN the legacy `secrets` table contains existing data, THE Credential_Store SHALL provide a data migration path that preserves existing encrypted values.

### Requirement 10: Monitor Credential Security

**User Story:** As a Pulse user, I want my monitor credentials encrypted and never exposed in API responses or logs, so that sensitive authentication data remains protected.

#### Acceptance Criteria

1. THE Credential_Store SHALL encrypt all secret values (tokens, passwords, header values) at rest using AES-256-GCM with the existing `PULSE_SECRET_KEY`.
2. THE Credential_Store SHALL support key rotation for monitor credentials using the existing `make rotate-key` mechanism.
3. THE Credential_Store SHALL sanitize log output to exclude decrypted credential values.
4. WHEN the API returns credential data, THE Credential_Store SHALL omit all plaintext secret fields from the response body.
5. THE Credential_Store SHALL decrypt credentials only at the moment of health-check execution within the Checker, keeping decrypted values in memory for the minimum duration required.
6. THE Credential_Store SHALL NOT expose any endpoint or mechanism to retrieve, decrypt, or display stored secret values after initial creation.

### Requirement 11: Monitor Credential API Endpoints

**User Story:** As a developer integrating with Pulse, I want RESTful API endpoints for managing monitor credentials, so that I can automate credential configuration.

#### Acceptance Criteria

1. THE Credential_Store SHALL expose `POST /api/v1/monitors/{id}/credentials` for creating a credential.
2. THE Credential_Store SHALL expose `GET /api/v1/monitors/{id}/credentials` for listing credentials of a monitor.
3. THE Credential_Store SHALL expose `PUT /api/v1/monitors/{id}/credentials/{credentialId}` for updating a credential.
4. THE Credential_Store SHALL expose `DELETE /api/v1/monitors/{id}/credentials/{credentialId}` for deleting a credential.
5. THE Credential_Store SHALL require authentication (JWT or API Token) for all credential endpoints.
6. THE Credential_Store SHALL validate that the authenticated user has access to the specified monitor before performing credential operations.

### Requirement 12: Frontend Monitor Credential Configuration

**User Story:** As a Pulse user, I want to configure authentication credentials within the monitor create/edit form, so that credential management is contextual to the monitor it belongs to.

#### Acceptance Criteria

1. WHEN a user creates or edits an HTTP or WebSocket monitor, THE Monitor_Config_Page SHALL display an "Authentication" section for managing credentials.
2. THE Monitor_Config_Page SHALL allow users to select an auth_type (Bearer Token, Basic Auth, or Custom Header) and provide the corresponding values.
3. WHEN a user saves a monitor with new credential values, THE Monitor_Config_Page SHALL create the credentials via the API and display the show-once creation modal.
4. THE Monitor_Config_Page SHALL display existing credentials as a list showing only metadata (auth_type, name, created_at), with options to replace or delete each entry.
5. THE Monitor_Config_Page SHALL NOT provide any reveal toggle or mechanism to display the secret values of existing credentials.
6. WHEN a user replaces an existing credential, THE Monitor_Config_Page SHALL treat the replacement as a new credential creation, displaying the new secret value exactly once in the creation modal.
7. WHILE a user is editing a TCP or UDP monitor, THE Monitor_Config_Page SHALL hide the "Authentication" section since these protocols do not support header-based auth.


### Requirement 13: Show-Once Secret Display Modal

**User Story:** As a Pulse user, I want a consistent modal experience when creating credentials or API tokens, so that I can securely copy my secret before it becomes permanently inaccessible.

#### Acceptance Criteria

1. WHEN a Monitor_Credential or API_Token is created successfully, THE Show_Once_Modal SHALL appear displaying the raw secret value in a read-only text field.
2. THE Show_Once_Modal SHALL provide a copy-to-clipboard button that copies the raw secret value to the system clipboard.
3. THE Show_Once_Modal SHALL display a prominent warning message stating that the secret value will not be shown again after dismissal.
4. WHEN the user clicks the dismiss or close action on the Show_Once_Modal, THE Show_Once_Modal SHALL clear the secret value from the component state and close.
5. THE Show_Once_Modal SHALL NOT allow the user to navigate away from the modal without explicitly dismissing it, ensuring the user acknowledges the one-time display.
6. IF the user copies the secret value using the copy button, THEN THE Show_Once_Modal SHALL display a confirmation indicator (e.g., "Copied!") for at least 2 seconds.
